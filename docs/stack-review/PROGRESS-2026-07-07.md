## The plan (three phases, from SERVER-DIRECTION.md)

**Thesis: the runtime supervises your consumers.** Crash-linked consumer
groups (no rebalances), producer-actionable backpressure (Defer/Reject in
the publish return, not a lag dashboard), resume-not-redeliver durable
processing, fork-at-hash replay, schema-in-bus, embedded zero-hop mode.

- **Phase 1 — "messages arrive"**: delivery pump + G4 writer, durable
  restart recovery, /metrics, auth table stakes → then 0.12.1 pin → A1-1
  bounded inboxes. Exit test: >96KB stream to a remote subscriber under
  20+ connections with credit-gated flow.
- **Phase 2 — activate the moats**: A1 complete, A3 routing/consumer-group
  wire, resume-not-redeliver surfaced, v2 explicit-credit acks.
- **Phase 3 — storage moat & reach**: fork-at-hash replay/forensics (anchors
  must be snapshot-named/branch-recorded per haematite `fork_at` semantics),
  TS WebSocket transport, embedded backends.

## Deferred (recorded, deliberate)

- Single Deliver frame > 4 MiB outbound buffer = connection teardown
  (spec-inherent; candidate follow-up: publish-side size cap).
- Deliver envelope fidelity is payload+schema+seq; causal metadata +
  publisher identity deferred (documented in delivery.rs).
- Greedy per-slice pump budget in HashMap order (latent while v1 = one
  subscription per dedicated connection); per-skipped-frame timeout re-arm
  on shared transports; pre-auth Ping allowed (slightly wider than the
  pre-merge gate — benign, unpinned either way).
- Cross-node pressure, live traffic-shadowing forks: explicitly not chased.
